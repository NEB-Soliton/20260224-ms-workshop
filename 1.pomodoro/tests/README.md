# テストドキュメント

## 概要

このドキュメントは、ポモドーロタイマーアプリケーションのテストスイートについて説明します。

## テスト統計

- **総テスト数**: 51
- **コードカバレッジ**: 97%
- **テスト実行時間**: 約1.5秒

## テスト構成

### 1. モデル層テスト (`test_models.py`)

データベースモデルの基本機能をテストします。

#### User モデル (4テスト)
- ユーザー作成
- ユーザー名の一意性制約
- リレーションシップ（セッション、バッジ）
- カスケード削除

#### PomodoroSession モデル (3テスト)
- セッション作成
- セッション完了
- ユーザーとのリレーションシップ

#### Badge モデル (2テスト)
- バッジ作成
- ユーザーとのリレーションシップ

### 2. サービス層テスト (`test_services.py`)

ビジネスロジックと状態遷移をテストします。

#### GamificationEngine (13テスト)

##### XP計算 (3テスト)
- ストリークなし: 基本XP（100）
- 3日ストリーク: +50ボーナス（合計150）
- 7日ストリーク: +100ボーナス（合計200）

##### レベル計算 (1テスト)
- レベル計算式: `level × 200 XP`
- レベル1: 0-199 XP
- レベル2: 200-599 XP
- レベル3: 600-1199 XP

##### ストリーク更新 (4テスト)
- 初回ポモドーロ: ストリーク1に設定
- 同日: ストリーク変更なし
- 連続日: ストリーク+1
- ストリーク途切れ: ストリーク1にリセット

##### バッジ授与 (5テスト)
- First Pomodoro: 1回完了時
- Dedicated: 10回完了時
- Master of Focus: 50回完了時
- Week Warrior: 7日ストリーク達成時
- 重複授与防止

#### PomodoroService (8テスト)

##### セッション開始 (2テスト)
- 正常系: セッション作成成功
- 異常系: 無効なユーザーID

##### セッション完了 (4テスト)
- 正常系: 完了成功（XP付与、バッジ授与、ユーザー統計更新）
- 異常系: 無効なセッションID
- 異常系: 既に完了済みのセッション
- 正常系: 複数セッションでストリーク構築

##### ユーザー統計取得 (2テスト)
- 正常系: 統計情報取得
- 異常系: 無効なユーザーID

### 3. API契約テスト (`test_api.py`)

REST APIエンドポイントの動作をテストします。

#### ユーザーAPI (8テスト)

##### POST /api/users
- 正常系: ユーザー作成成功
- 異常系: ユーザー名なし（400 Bad Request）
- 異常系: ユーザー名重複（409 Conflict）
- 異常系: 無効なJSON（400 Bad Request）

##### GET /api/users/{user_id}
- 正常系: ユーザー詳細取得
- 異常系: ユーザーが存在しない（404 Not Found）

##### GET /api/users/{user_id}/stats
- 正常系: ユーザー統計取得
- 異常系: ユーザーが存在しない（404 Not Found）

#### セッションAPI (8テスト)

##### POST /api/sessions/start
- 正常系: セッション開始成功
- 異常系: user_idなし（400 Bad Request）
- 異常系: durationなし（400 Bad Request）
- 異常系: 無効なduration - 負の値、ゼロ、非整数（400 Bad Request）
- 異常系: ユーザーが存在しない（404 Not Found）

##### POST /api/sessions/{session_id}/complete
- 正常系: セッション完了成功
- 異常系: セッションが存在しない（404 Not Found）
- 異常系: 既に完了済み（404 Not Found）

#### バッジAPI (3テスト)

##### GET /api/users/{user_id}/badges
- 正常系: バッジ一覧取得（複数）
- 正常系: バッジ一覧取得（空）
- 異常系: ユーザーが存在しない（404 Not Found）

#### 統合フローテスト (2テスト)
- 完全なポモドーロワークフロー（ユーザー作成→セッション開始→完了→統計確認→バッジ確認）
- 複数セッション完了とバッジ獲得

## テスト実行方法

### すべてのテストを実行

```bash
cd 1.pomodoro
python -m pytest tests/ -v
```

### カバレッジレポート付きで実行

```bash
python -m pytest tests/ -v --cov=. --cov-report=html
```

カバレッジレポートは `htmlcov/index.html` で確認できます。

### 特定のテストファイルのみ実行

```bash
python -m pytest tests/test_models.py -v
python -m pytest tests/test_services.py -v
python -m pytest tests/test_api.py -v
```

### 特定のテストクラスのみ実行

```bash
python -m pytest tests/test_services.py::TestGamificationEngine -v
```

### 特定のテスト関数のみ実行

```bash
python -m pytest tests/test_api.py::TestUserEndpoints::test_create_user_success -v
```

## テスト設計原則

### 1. **AAA パターン**
すべてのテストは以下の構造に従います：
- **Arrange** (準備): テストデータとコンテキストのセットアップ
- **Act** (実行): テスト対象の機能を実行
- **Assert** (検証): 期待される結果を検証

### 2. **独立性**
各テストは独立して実行可能で、他のテストの実行順序や結果に依存しません。

### 3. **テストデータベース**
- インメモリSQLiteデータベースを使用
- 各テストの前後でデータベースを初期化・クリーンアップ

### 4. **フィクスチャの活用**
`conftest.py`で共通のテストフィクスチャを定義：
- `app`: テストアプリケーションインスタンス
- `client`: テストクライアント（APIテスト用）
- `test_user`: テスト用ユーザー

## トラブルシューティング

### テストが失敗する場合

1. **依存関係の確認**
```bash
pip install -r requirements.txt
```

2. **データベースの問題**
- インメモリデータベースを使用しているため、永続的な問題は発生しません
- テスト間でデータが残る場合は、フィクスチャの設定を確認

3. **詳細なエラー情報**
```bash
python -m pytest tests/ -v --tb=long
```
